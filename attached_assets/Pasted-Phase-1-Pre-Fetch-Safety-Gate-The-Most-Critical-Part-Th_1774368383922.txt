Phase 1 — Pre-Fetch Safety Gate (The Most Critical Part)
This is the answer to "how do I ensure I'm not installing fake packages."

1A. Trusted Package Name List

On backend startup, fetch the top 8,000 PyPI packages from https://hugovk.github.io/top-pypi-packages/top-pypi-packages-30-days.min.json
Sign the entire list with ML-DSA-65 + Ed25519 immediately after fetching
Cache with 24-hour TTL, re-fetch and re-sign on expiry
Before every name check, verify the ML-DSA signature on the cached list — if tampered, block the check entirely
1B. Name Verification Engine (runs before any PyPI call)

Exact match → VERIFIED (green, proceed)
Levenshtein distance 1 to any top-8K package → LIKELY TYPOSQUAT (red, block with modal)
Levenshtein distance 2 → SUSPICIOUS (yellow, warn with modal)
Homoglyph normalization first: 0→o, 1→l, rn→m, vv→w, then run distance check
Padding patterns: python-X, X-lib, py-X, get-X where X is a known package → SUSPICIOUS
Not in top-8K but no near-matches → UNKNOWN (grey, proceed with warning)
1C. PQC-Signed Name Attestation

Every verdict (including VERIFIED) is serialized to canonical JSON and signed with ML-DSA-65 + Ed25519
Output: SignedNameAttestation with the verdict, confidence score, nearest match, edit distance, and both signatures
This happens before fetch_package() is called — it is a hard gate
1D. User Flow for Blocked Names

VERIFIED: package card loads immediately with a green shield badge
SUSPICIOUS/TYPOSQUAT: modal appears — "This looks like a typosquat of [X]. Do you want to proceed anyway? This decision will be recorded in your signed attestation." User clicks confirm or cancel.
UNKNOWN: soft warning on the card, user can proceed
If cancelled: package is not added, no fetch happens
Phase 2 — Package Risk Scoring (Runs After Fetch)
Once the name is verified and metadata is fetched, compute a 0–100 risk score per package from these signals:

Signal 1 — CVE severity (0–40 points of risk)

CRITICAL CVE: +40, HIGH: +25, MEDIUM: +10, LOW: +5
Signal 2 — Package age and activity (0–20 points)

First release less than 6 months ago: +15
No release in over 3 years: +10
Only 1 release ever: +10
Signal 3 — Maintainer count (0–15 points)

Single maintainer: +15 (single point of failure / account takeover risk)
2–3 maintainers: +5
Signal 4 — Download volume (0–15 points)

Under 1,000 downloads/month: +15
Under 10,000: +8
Signal 5 — Name check result (0–10 points)

UNKNOWN: +5
SUSPICIOUS (user confirmed): +10
Display: Each package card shows a color-coded risk score — green (0–25), yellow (26–60), red (61–100). The full project gets an aggregate risk score. Sort packages by risk score by default.

Phase 3 — Static Source Scan (Runs on sdist)
For packages where a source distribution exists:

Download the sdist tar.gz from PyPI (do not execute it)
Extract and scan setup.py, setup.cfg, pyproject.toml, and all __init__.py files for:
exec(base64.b64decode( — obfuscated execution
subprocess.call, os.system, os.popen in setup files — runs on install
urllib, requests, socket in setup files — network calls on install
Patterns like AWS_, SSH_, .ssh/id_rsa, GITHUB_TOKEN — credential harvesting
import ctypes, import cffi in setup files — native code injection
Return a list of StaticFinding objects: {pattern, file, line_number, severity, description}
Sign the scan result with ML-DSA-65 + Ed25519
Show findings on the package card in a collapsible "Static Analysis" section

Phase 5 — Quantum Threat Simulator Tab
A tab called "Quantum Threat" showing:

A side-by-side comparison: classical ECDSA vs ML-DSA-65
Grover's algorithm impact on SHA256: halves security from 256-bit to 128-bit — still safe
Shor's algorithm impact on ECDSA/RSA: breaks it entirely — why classical signatures fail
A live demo: take the ECDSA public key from the current session and show that a quantum attacker with enough qubits could recover the private key. Then show that ML-DSA-65 is based on Module Learning With Errors — no known quantum algorithm breaks it
Show the lattice visualization: a 2D grid where the shortest vector problem is illustrated — the mathematical foundation of CRYSTALS-Dilithium
Phase 6 — License Compliance Engine
For each package in the resolved list:

Map the license string to an SPDX identifier
Flag incompatibilities:
GPL-3.0 in a project with MIT components: copyleft contamination warning
AGPL-3.0 in any project: requires open-sourcing even SaaS backends
Multiple licenses on one package: ambiguity warning
Show a license compatibility matrix across all packages
Sign the compliance report with ML-DSA-65
